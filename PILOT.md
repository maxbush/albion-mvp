# 🚀 Демо-пилот ALBION — сценарий «Неявка ученика» на реальных TG-аккаунтах

> Цель пилота: на живых Telegram-аккаунтах владельцев показать MVP-сценарий из
> «Видения ALBION» — **неявка ученика → уведомление родителя → кнопка
> подтверждения → эскалация координатору только при необходимости**
> (management by exception).

Пилот доказывает клиенту главное: система **объединяет всех участников процесса
в едином автоматизированном контуре** и снимает боль №1 — ручной поиск
отсутствующих учеников (до 50% рабочего дня координатора).

---

## 🧭 Как это работает (端到端)

```
Репетитор (tutor)        Координатор ALBION AI            Родитель (parent)
      │                          │                              │
      │  отмечает неявку         │                              │
      ├─────────────────────────►│                              │
      │                          │  уведомляет и ждёт ответа    │
      │                          ├─────────────────────────────►│  «✅ Всё в порядке»
      │                          │                              │
      │                          │◄─────────────────────────────┤  нажал кнопку
      │                          │  инцидент закрыт             │
      │                          │                              │
      │                  (если тишина N мин)                    │
      │                          │  эскалация координатору      │
      │                          ├─────────────► coordinator разбирает ТОЛЬКО критическое
```

Участники играют **реальные роли**, назначенные командой `/role` по их
Telegram-аккаунтам.

## ✅ Предварительные условия

1. **Python 3.11+** и зависимости:
   ```bash
   python -m venv .venv && source .venv/bin/activate
   pip install -r requirements.txt
   ```
2. **Тестовый TG-бот**: создать у [@BotFather](https://t.me/BotFather) → получить токен.
3. **2–4 Telegram-аккаунта владельцев**, которые сыграют роли
   (координатор / репетитор / родитель). Удобно с разных телефонов или Telegram X.
4. Доступ к **MeritHub API** (Admin Console → API): `MERITHUB_CLIENT_ID` +
   `MERITHUB_CLIENT_SECRET` + сгенерированный на странице интеграций секрет вебхуков.
   Без них бот работает на mock-данных (сценарий всё равно прогоняется на реальных TG).

## ⚙️ Шаг 1. Настройка `.env`

Скопируйте `.env.example` → `.env` и заполните:

```dotenv
TELEGRAM_BOT_TOKEN=<токен от BotFather>

# TG ID владельцев-админов через запятую (узнать свой ID: /whoami в боте).
# Эти аккаунты смогут раздавать роли (/role) и запускать пилот (/pilot_absent).
ALBION_ADMIN_TELEGRAM_IDS=<ваш TG ID>

# Для ЖИВОГО демо — короткие тайминги (иначе 5/15 минут как в проде):
ALBION_NOTIFY_PARENT_DELAY_MIN=1
ALBION_ESCALATE_DELAY_MIN=1
ALBION_PILOT_STUDENT_NAME=Миша Иванов

# MeritHub (реальная интеграция, OAuth2+JWT) — Client ID/Secret из Admin Console → API.
# Секрет вебхуков — кнопка «Generate API Secret» на странице интеграций. НЕ в чат.
MERITHUB_CLIENT_ID=
MERITHUB_CLIENT_SECRET=
MERITHUB_WEBHOOK_SECRET=

# Демо-сид с захардкоженными пользователями для пилота НЕ нужен:
ALBION_DEMO_MODE=false
```

> **Как узнать свой TG ID:** запустите бота, напишите `/whoami` — бот покажет
> `TG ID: 123456789`. Впишите нужные ID в `ALBION_ADMIN_TELEGRAM_IDS`.

## 👥 Шаг 3. Раздача ролей владельцам

Каждый владелец пишет боту `/start`, затем `/whoami` и сообщает свой **TG ID** админу.

Админ раздаёт роли:

```
/role 111111111 coordinator     # Мария — координатор
/role 222222222 tutor           # Анна — репетитор
/role 333333333 parent          # Ольга — родитель Миши
```

Контроль состава:

```
/roles
# 👥 Участники и роли
# 👨‍💼 `111111111` @maria — coordinator ★ — Мария
# 🧑‍🏫 `222222222` @anna — tutor — Анна
# 👨‍👩‍👦 `333333333` @olga — parent — Ольга
```

| Роль | Что делает в пилоте |
|------|---------------------|
| `coordinator` 👨‍💼 | Получает эскалации и системные алерты (DLQ). Разбирает только критические случаи. |
| `tutor` 🧑‍🏫 | Отмечает неявку ученика (`/absent <ID>` / `/pilot_absent`). |
| `parent` 👨‍👩‍👦 | Получает уведомление о неявке с кнопкой «✅ Всё в порядке». |
| `student` 🎓 | Зарезервирована (ученики живут в MeritHub). |

## 🎬 Шаг 5. Прогон сценария неявки

Админ (или репетитор) запускает:

```
/pilot_absent
```

Что произойдёт:

1. **~через 10 секунд** аккаунт-`parent` получит сообщение:
   > 👋 Здравствуйте!
   > Миша Иванов отсутствовал(а) на занятии (ID: pilot_lesson_1).
   > Всё ли в порядке?
   >
   > **[ ✅ Всё в порядке ]**
2. **Вариант А (счастливый путь):** родитель жмёт «✅ Всё в порядке» →
   инцидент закрывается, кнопки пропадают, эскалация отменяется. Координатор
   **не отвлекается** — management by exception в действии.
3. **Вариант Б (эскалация):** родитель молчит `ALBION_ESCALATE_DELAY_MIN` минут →
   `coordinator` получает 🚨 эскалацию и разбирает случай вручную.

Запустите оба варианта, чтобы показать клиенту полную картину.

## 🔁 Повторный прогон

`/pilot_absent` можно запускать многократно — каждый запуск создаёт новую
ситуацию. Статистику (сколько закрыто, среднее время реакции) видно через
демо-отчёт координатора, а свежие инциденты — в SQLite (`albion.db`).

## 🛰 MeritHub по-настоящему: OAuth2 + вебхуки → авто-неявки

Интеграция боевая, по официальной документации MeritHub.

### 1. Credentials (Admin Console → API)
```dotenv
MERITHUB_CLIENT_ID=...
MERITHUB_CLIENT_SECRET=...   # НЕ в чат
```
Клиент сам получает access token по **OAuth2** (JWT HS256, подпись = `CLIENT_SECRET`),
кеширует его на 55 минут и обновляет по 401. Хосты по умолчанию — из док
(`serviceaccount1.meritgraph.com`, `class1.meritgraph.com`, `live.merithub.com`).

> MeritHub **не отдаёт** список юзеров/классов через API — возвращённые `UserId`/
> `ClassId` храним сами (таблицы `merithub_students`, `merithub_enrollments`).

### 2. Заводим учеников и связываем с TG родителя
```
/mh_user <clientUserId> <TG_родителя> <Имя ученика>
```
Создаёт ученика в MeritHub (role `M`) и сохраняет маппинг
`clientUserId ↔ merithubUserId ↔ TG родителя`. Родитель регистрируется в системе
(иначе уведомление ушло бы в эскалацию). Список привязок: `/mh_students`.

### 3. Зачисляем в класс (иначе неявку не посчитать)
```
/mh_enroll <classId> <clientUserId1> [clientUserId2 ...]
```

### 3b. Или одной командой — создать класс и зачислить (`/mh_schedule`)
Если класс заводите через нас (а не в UI MeritHub), `/mh_enroll` не нужен:
```
/mh_schedule <tutorCuid> <startRFC3339> <durationMin> <studentCuid> [...]
```
Команда сама создаёт класс в MeritHub (через репетитора, заведённого `/mh_tutor`),
зачисляет репетитора (`commonHostLink`) и учеников (`commonParticipantLink`) и
**сохраняет зачисление** — чтобы вебхук `attendance` потом посчитал неявки.
Возвращает `classId` и ссылку на комнату репетитора. Это замкнутый путь: после него
живой урок → вебхук → авто-уведомление родителя работают без UI MeritHub.

> Репетиторов (role `C`) заводите командой `/mh_tutor <clientUserId> <имя>`.

### 4. Страница интеграций MeritHub (со скрина)
- **Redirect Url** — оставить пустым.
- **Webhook Url** — `<публичный https>/merithub/webhook` (туннель/VPS; localhost нельзя).
- **Content Webhook Url** — оставить пустым.
- **Чекбоксы**: включить **Attendance** (главное), **Session Status**,
  **Users Added in Organization**, **Users Added in Session**; остальное выкл.
- **Generate API Secret** — это, скорее всего, `CLIENT_SECRET` для JWT (→ в
  `MERITHUB_CLIENT_SECRET`). Сам MeritHub **не подписывает** вебхук-пинги по доке,
  поэтому `MERITHUB_WEBHOOK_SECRET` задавайте лишь как доп. проверку на случай,
  если пинги придут с заголовком подписи (иначе приёмник принимает пинги и так).

### 5. Запуск приёмника вебхуков + туннель
```bash
python -m src.main                                   # терм 1: бот
uvicorn src.api.webhook:app --host 0.0.0.0 --port 8000   # терм 2: вебхуки (та же SQLite, WAL)
cloudflared tunnel --url http://localhost:8000       # терм 3: публичный https (или ngrok http 8000)
```
Полученный https-URL + `/merithub/webhook` → в MeritHub **Webhook Url**.

### 6. Авто-сценарий (без ручного /absent!)
По окончании урока MeritHub шлёт `requestType: "attendance"` со списком
присутствовавших (`userId` + `totalTime`). Приёмник считает
**зачисленные − присутствовавшие = отсутствующие** и для каждого запускает
`trigger_absence` → родитель на реальном TG получает уведомление с кнопкой
«✅ Всё в порядке»; при молчании — эскалация координатору. Это и есть
«management by exception» из Видения — полностью автоматически.

Дискриминатор событий — поле **`requestType`** (`classStatus`/`attendance`/`recording`…).
Сырой payload всегда сохраняется — посмотреть: `/mh_events`.

> Ручной `/pilot_absent` по-прежнему работает — вебхук-путь его дополняет.

## 🧩 Что реально, а что mock (честно)

| Компонент | Статус в пилоте |
|-----------|-----------------|
| Telegram-бот, команды, inline-кнопки | ✅ реальные |
| Роли владельцев (`/role`, `/roles`, `/whoami`) | ✅ реальные, в SQLite |
| Сценарий неявки: инцидент → уведомление → кнопка → эскалация | ✅ реальный, на живых TG |
| SQLite-шедулер, DLQ, идемпотентность, kill switch | ✅ реальные |
| Данные учеников/уроков | ✅ реальные через MeritHub API + маппинги `/mh_user`/`/mh_tutor`/`/mh_schedule`/`/mh_enroll` |
| MeritHub API | ✅ реальный клиент OAuth2+JWT (`merithub_client.py`): юзеры/классы/ссылки |
| Авто-неявки по вебхуку | ✅ `requestType=attendance` → `trigger_absence` → уведомление родителя |
| Airtable / Xero | ⏳ mock, real — по roadmap |

Когда получим доступ к **MeritHub API**, ученики и расписание будут браться
оттуда (клиент сам заведёт ~10 учеников по API), а сценарий неявки станет
полностью боевым. Переключение — одна пара переменных окружения
(`src/integrations/factory.py`), без правки бизнес-логики (принцип **Vendor Agnostic**).

## 🛠 Troubleshooting

| Проблема | Решение |
|----------|---------|
| Бот не отвечает | Проверьте `TELEGRAM_BOT_TOKEN` и что запущен `python -m src.main` (polling). |
| `/role` отвечает «⛔ Только владелец» | Впишите свой TG ID в `ALBION_ADMIN_TELEGRAM_IDS` и перезапустите бота. |
| `/pilot_seed` пишет «❌ Не хватает ролей» | Назначьте хотя бы одного `parent` и одного `coordinator` через `/role`. |
| Родитель не получил уведомление | Аккаунт-родитель должен быть зарегистрирован (владелец написал `/start`) и иметь роль `parent`. |
| Уведомления не уходят вовсе | Проверьте ` /status` → Kill Switch должен быть `2` (полностью). |
| Хочу сбросить данные | Удалите файл `albion.db` и перезапустите бота. |
| Родитель не получил авто-уведомление по вебхуку | `/mh_events` — пришёл ли `attendance`; `/mh_students` и `/mh_enroll` — есть ли зачисление и TG родителя. |
| MeritHub ретраит вебхук / 401 | `MERITHUB_WEBHOOK_SECRET` должен совпадать со сгенерированным; сырой payload + заголовки видны в `/mh_events`. |

## 🎯 Критерии успеха демо для клиента

- [ ] Все владельцы зашли в бота и получили роли через `/role`.
- [ ] `/pilot_seed` показывает «✅ Пилот готов».
- [ ] При `/pilot_absent` родитель **на реальном аккаунте** получает уведомление с кнопкой.
- [ ] Нажатие «✅ Всё в порядке» закрывает инцидент без участия координатора.
- [ ] При молчании родителя координатор получает эскалацию (management by exception).
- [ ] Перезапуск бота не теряет состояние (инциденты/задачи в SQLite).
